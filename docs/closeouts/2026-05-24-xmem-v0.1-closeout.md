## What Is Done

- `xmem context` returns layered symbolic packets with `node_id`, `memory_layer`, `source_ref`, `source_path`, and drilldown hints.
- `xmem preflight` returns development-start guard packets and supports `--fields domain=... service=... repo=... task=... mode=...` for cleaner hook queries.
- Project Wiki export, Project Wiki pending inbox, Issue Record bug-pattern export, xmem outbox, local cards, map refs, and codegraph refs are imported into the generated registry.
- Project Wiki pending rows are searchable but hint-only; they do not override verified truth.
- Issue Record remains the owner for bug patterns. The latest verified imported guard is `issue-pattern.scmp-helper-output-secret-redaction`.
- `xmem gain` now explains confidence, rough-vs-confirmed savings, top queries, top cards, and card detail; `xmem gain card <id>` drills into a noisy or high-frequency card.
- `xmem suppress` records ranking feedback for true-but-irrelevant hits without corrupting source truth.
- `xmem doctor` is the maintenance view for registry, source exports, local-card portability, backup health, outbox, and repo registration.
- `xmem-backup` is configured and pushed as the unified mirror/backup repo.

## Boundaries Future Agents Must Preserve

- Do not turn xmem into a fourth truth DB.
- Do not silently promote Project Wiki pending rows, single-incident bug patterns, generated indexes, or runtime current state to `verified`.
- Project/entity truth belongs in Project Wiki; bug truth belongs in Issue Record; xmem stores compact cards, imports, source freshness, routing refs, and ranking feedback.
- Dynamic service state, current branch, latest deploy, pod state, and SCMP current status must be live-verified before acting.
- Raw command output, large JSON, Feishu readback payloads, and broad `rg` results should be stored as evidence paths and surfaced to agents as compact summaries.
- `validation_service` means candidate traffic target before cutover, not generic test environment.
